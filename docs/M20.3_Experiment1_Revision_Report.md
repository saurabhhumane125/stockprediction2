# Experiment 1: Revision Report

## 1. INFERENCE ENGINE CHANGE ANALYSIS

1. **Why was this modification required?**
   The decoder interface `BaseInferenceDecoder.decode(self, raw_prediction: float, current_features: np.ndarray, ...)` explicitly requires the current feature state to perform dynamic decoding (such as un-scaling volatility or converting percentage predictions back to absolute prices). The previous implementation in `ProductionInferenceEngine.predict()` was hardcoded to pass `None` for this argument.
   
2. **Was this caused by an existing architectural limitation or the implementation design itself?**
   It was caused by an **existing architectural limitation** (a defect). The original legacy classification pipeline did not require feature state to decode `0` or `1`, so the engine was never wired to fulfill the decoder's contract.

3. **Can the decoder work WITHOUT modifying ProductionInferenceEngine?**
   No. Any mathematical decoder that reverses normalisation relative to a dynamic current state (e.g., current volatility, current price, ATR) fundamentally requires that state.

4. **If no, provide technical evidence proving that the engine modification is mandatory for all future target strategies:**
   Any future regression target (e.g., Return %, Z-Score, Volatility-Scaled) will output a dimensionless or scaled float (e.g., `2.5`). To convert `2.5` back into an actionable trading prediction (e.g., "Expected Price = $150"), the decoder must have access to the raw $P_t$ (current price) or $\sigma_t$ (current volatility). Passing `None` breaks the architectural contract of the Target Strategy Framework.

5. **If the engine modification is retained, justify why it is an infrastructure correction rather than an experiment-specific change:**
   It is an **Approved Infrastructure Fix**. It restores the intended contract between the Inference Engine and the Decoder registries, enabling all future Target Strategies to function correctly. It has zero impact on the legacy decoder (which simply ignores the argument).

## 2. DECODER DESIGN REVIEW

**Approach A (Recompute Volatility):**
The decoder extracts raw `close` prices from the `current_features` sequence and recalculates the 20-day standard deviation identically to the training logic.
- *Determinism:* High, as it executes the exact same mathematical formula.
- *Stability / Drift:* Zero floating-point drift because it relies on raw unscaled prices.
- *Risk:* Fails critically if `TrainingConfig.SEQUENCE_LENGTH < 20`, as there won't be enough historical data in the inference window to compute a 20-day rolling metric.

**Approach B (Reuse Engineered Volatility Feature):**
The decoder extracts a pre-computed `vol_20d` scalar directly from the feature vector.
- *Determinism:* Perfect (reuses the exact scalar).
- *Stability / Drift:* Low drift, provided the feature isn't subjected to destructive non-linear scaling before the decoder reads it.
- *Risk:* Tightly couples the Target Framework to the Feature Engineering pipeline. It requires the feature engineering team to maintain a specific `vol_20d` feature indefinitely.

**Recommendation: Approach A (Recompute Volatility)**
*Justification:* Experiment 1 is strictly a single-variable experiment. Forcing the feature engineering pipeline to add and maintain a new `vol_20d` feature violates the single-variable constraint. Approach A preserves absolute architectural decoupling: the Target Strategy is entirely self-contained. To mitigate the risk of short sequence lengths, the decoder must strictly validate that `len(current_features) >= 20` and raise an explicit `ConfigurationError` if violated.

## 3. TARGET VALIDATION REVIEW

The previous statistical validation was scientifically invalid because it compared a discrete binary classification target (`[0, 1]`) against a continuous continuous regression target (`[-35.0, 19.0]`). 

**Redesigned Validation Protocol:**
The validation script must be completely rewritten to compare **Raw Return Target (Return_5d)** against **Volatility Adjusted Return Target**. 

Comparing these two continuous targets yields legitimate scientific conclusions:
- **Standard Deviation & Variance:** A reduction here proves that the new target successfully suppresses extreme market volatility, making the loss landscape smoother.
- **Skewness & Kurtosis:** A reduction in kurtosis proves that "fat tails" (extreme outliers) have been successfully normalized, reducing the risk of gradient explosion during training.
- **Minimum, Maximum, Range:** Proves numerical bounding. A raw return could theoretically be +200% during a short squeeze; the volatility-adjusted target should pull this extreme outlier closer to a standardized range (e.g., `+3.0`).

No claims of "model improvement" can be made from this script—it only proves that the mathematical distribution of the training data behaves as hypothesized.

## 6. FINAL DECISION

**REQUIRES ARCHITECTURAL REVISION**

*Technical Justification:* While the framework modifications were justified as infrastructure fixes, the statistical validation protocol was fundamentally flawed, comparing apples (classes) to oranges (continuous scalars). A flawed validation script compromises the scientific integrity of the ML experiment. Furthermore, the decoder currently lacks robust defensive programming against configuration changes (e.g., `SEQUENCE_LENGTH` reductions). These two revisions must be implemented before moving to production GPU training.
