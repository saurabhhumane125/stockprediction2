# Final Implementation Plan: Experiment 1 (Volatility-Adjusted Target)

## 1. VOLATILITY DEFINITION
The target will be a Simple Volatility-Scaled Return. The implementation will be strictly deterministic and reproducible.

**Mathematical Definition:**
`Target = Return_5d / Max(Vol_20d, Epsilon)`

**Parameters:**
- **Return_5d:** `(Close_{t+5} - Close_t) / Close_t`
- **Vol_20d:** Rolling standard deviation of 1-day returns over the preceding 20 trading days (`t-19` to `t` inclusive).
- **Standard deviation definition:** Sample standard deviation (`ddof=1`).
- **Rolling window length:** 20.
- **min_periods:** 20 (strict enforcement; missing values yield `NaN`).
- **NaN handling policy:** Any row where either `Return_5d` or `Vol_20d` evaluates to `NaN` must be dropped immediately (`df.dropna(subset=[target_col])`).
- **Zero volatility handling / Numerical stability policy:** To prevent divide-by-zero or massively inflated targets during flat-price periods (e.g., trading halts), the denominator will be bounded. Epsilon (`1e-8`) will be enforced using `np.clip(vol_20d, 1e-8, np.inf)`. If an asset has 0 volatility for a month, the target will effectively be bounded, though in practice such assets typically have `NaN` returns anyway.

## 2. DATASET VERSIONING
We will **not** introduce a secondary naming convention (e.g., `EXP_M20`). We will strictly utilize the existing semantic versioning sequence of the production datasets.
- **Baseline Dataset Version:** `CORE/v1.0`
- **Experimental Dataset Version:** `CORE/v2.0` (Assuming v1.0 was the legacy baseline).
- **Mechanism:** The Target Strategy Framework generates targets dynamically at build-time. We will increment the `dataset_version` in `TrainingConfig` to `CORE/v2.0` and set the `strategy_name = "volatility_adjusted"`. The pipeline will generate the new dataset entirely independently into the `CORE/v2.0` directory without mutating `CORE/v1.0`.

## 3. DECODER DESIGN
The `VolatilityAdjustedInferenceDecoder` must map the neural network's scaled scalar back to a real expected return. 

`Predicted_Return_5d = Raw_Prediction * Vol_20d`

**Priority Logic to prevent Numerical Drift:**
1. **Reuse Engineered Features:** The decoder will first attempt to extract the volatility from the incoming `current_features` dictionary/series. Assuming the feature generator produces a column named `return_std_20d` (or equivalent), the decoder will extract this unscaled value directly.
2. **Deterministic Fallback Calculation:** If the feature does not exist, the decoder must calculate it identically. It will extract the raw `close` prices sequence provided in the inference context, compute the 1-day returns, and apply the exact same `std(ddof=1)` over the 20 steps. 
**Drift Prevention:** By relying on the unscaled historical raw features (or exact historical close prices) passed during inference, the decoder avoids any inverse-transform float-precision loss from the `StandardScaler`.

## 4. PRE-TRAINING VALIDATION
Before launching GPU training, a strict statistical validation script will execute, comparing `CORE/v2.0` (Experimental) against `CORE/v1.0` (Baseline).

**Metrics Compared:**
- Mean, Median, Minimum, Maximum
- Standard Deviation, Variance
- Skewness, Kurtosis
- NaN count, Outlier percentage (e.g., values > 3 std devs from mean)
- Histogram distributions (visual overlay)
- Target range limits

**Acceptable Differences (Success Thresholds):**
- **Mean:** The new target mean must remain centered near 0.
- **Standard Deviation:** Must be constrained (expected range ~0.5 to 3.0), proving heteroskedasticity reduction.
- **Skewness & Kurtosis:** Must be strictly lower (closer to a normal distribution) than the raw return baseline.
- **Outliers:** The percentage of extreme targets (>3 sigma) must be significantly reduced.

## 5. EXPERIMENT ACCEPTANCE CRITERIA
This experiment will only be permanently promoted to the baseline if it strictly satisfies these objective criteria:

- **Infrastructure Success:** The E2E Pipeline (`TensorBuilder`, `Trainer`, `Evaluator`, `RegistryManager`) completes successfully without any runtime exceptions or NaN crashes.
- **Training Success:** The model's loss function successfully converges (val_loss decreases by at least 5% from epoch 1) without gradient explosion.
- **Inference Success:** The `TargetManager` resolves the decoder in an isolated inference test, and the decoder successfully reverses the mathematical prediction into an absolute return format.
- **Registry Success:** Model metadata correctly logs `strategy_name: volatility_adjusted`, and artifacts are safely serialized.
- **Model Quality Comparison:** The experimental model must strictly outperform the legacy baseline on at least 3 of the predefined evaluation metrics (see Section 6).
- **Rollback Conditions:** If any of the above fail, the experiment is discarded. Rollback is executed purely via configuration: setting `TrainingConfig.dataset_version = "CORE/v1.0"` and `strategy_name = "legacy"`.

## 6. METRIC COMPARISON
The `ProductionEvaluator` will compare the new model's decoded outputs directly against the baseline model's decoded outputs on the exact same test-set chronological split.

**Objective Comparison Metrics:**
1. **RMSE (Root Mean Squared Error):** On the *decoded* returns.
2. **MAE (Mean Absolute Error):** On the *decoded* returns.
3. **R² (Coefficient of Determination):** Assessing explained variance.
4. **Information Coefficient (IC):** Spearman Rank Correlation between predicted returns and actual returns (Critical for sorting trading signals).
5. **Prediction Variance:** Standard deviation of predictions.
6. **Prediction Distribution:** KS-Test against actual test-set return distribution.
7. **Residual Distribution:** Assessing normality of error margins.
8. **Training Stability:** Variance of training loss over the final 5 epochs.
9. **Best Epoch / Early Stopping:** Analyzing if the model converges faster (fewer epochs to minimum val_loss).

## 7. RISK REVIEW
1. **Risk:** Epsilon Addition causing inflated targets.
   - *Resolution:* By clipping volatility to `1e-8` minimally, a 0 volatility state implies a flat line. A flat line will have a 0 numerator (0 return). `0 / 1e-8 = 0`. The target safely resolves to `0.0`.
2. **Risk:** Look-ahead Bias in Volatility Calculation.
   - *Resolution:* The calculation relies explicitly on `[t-19, t]`. The shift operation `shift(-5)` is applied strictly to the numerator. The denominator is unaware of future prices.
3. **Risk:** Decoder Unscaling Crash due to Missing Features.
   - *Resolution:* The decoder enforces a strict schema check. If `current_features` lacks both the engineered volatility column AND the raw close price history necessary to compute it, the decoder will raise a `ValueError` rather than silently failing or outputting garbage. 
4. **Risk:** Backward Compatibility Breaking.
   - *Resolution:* Legacy models run simultaneously will have `"legacy"` in their manifest and route entirely around this new code. Zero risk to existing artifacts.
