# M19.2 Target Strategy Framework Design

==========================================================
## SECTION 1 — CURRENT ARCHITECTURE
==========================================================
The current target generation flow operates as a centralized, imperative sequence:

1. **Dataset Generation** (`generate_datasets.py`): Initiates the dataset build process and passes a unified `TrainingConfig`.
2. **TargetFactory** (`target_factory.py`): Contains hardcoded `if/elif/else` blocks testing `task_type` (Binary, Multiclass, Regression, Multi-Output) and `target_type` (LOG_RETURN, PRICE, RETURN). It calculates the mathematical target array and drops tail NaNs.
3. **TensorBuilder** (`builder.py`): Calls `TargetFactory.generate(df, config)`, extracts the resulting `target_cols` as NumPy arrays, and exports them to `.npz` files (e.g., `y_train.npy`).
4. **Metadata** (`metadata.py`): Queries `TargetFactory.get_target_cols` to document the exact target column names (e.g., `"return_5d"`) in `metadata.json`.
5. **Training** (`training_pipeline.py`): `TimeSeriesDataset` blindly loads the `y` array from the tensor file. It does not know or care about the mathematical definition of the target (treating it simply as a `float32` objective for MSE).
6. **Inference Engine** (`engine.py`): Predicts a raw numerical scalar. It interprets the prediction type strictly using `task_type` from `metadata.json` (e.g., mapping class indices for classification, but outputting raw scalars for regression).
7. **Registry** (`manager.py`): Stores the resulting model and `metadata.json` with no explicit mathematical target schema.

**Where the target enters the pipeline**: The target logic is fully injected and locked at the `TargetFactory` step during dataset compilation.

==========================================================
## SECTION 2 — TARGET STRATEGY DESIGN
==========================================================
To eliminate hardcoded `if/else` chains and support extensible target engineering, we will implement the **Strategy Pattern** paired with a **Dynamic Registry**.

### 1. Abstract Strategy Interface (`BaseTargetStrategy`)
All future targets must implement a unified interface:
```python
class BaseTargetStrategy(ABC):
    @property
    @abstractmethod
    def strategy_name(self) -> str: pass

    @abstractmethod
    def get_target_cols(self, config: TargetConfig) -> List[str]: pass

    @abstractmethod
    def generate(self, df: pd.DataFrame, config: TargetConfig) -> Tuple[pd.DataFrame, List[str]]: pass

    @abstractmethod
    def post_process_inference(self, raw_prediction: float, current_features: pd.Series) -> float:
        """Transforms raw model output back into business domain logic (e.g., un-scaling volatility)."""
        pass
```

### 2. Strategy Registry
A central registry (`TargetStrategyRegistry`) that dynamically loads and caches registered strategies (e.g., `RawReturnStrategy`, `VolatilityAdjustedStrategy`, `BinaryDirectionStrategy`).

### 3. Configuration Driven
`TargetConfig` will introduce a new field: `strategy_name: str`. The `TensorBuilder` will ask the registry for the strategy matching `strategy_name` and execute it. 

==========================================================
## SECTION 3 — BACKWARD COMPATIBILITY
==========================================================
**Guaranteeing Existing Models Work**:
1. **Legacy Fallback**: If `TargetConfig.strategy_name` is missing (as it is in all current configs), the `TargetStrategyRegistry` will default to a new `LegacyTargetStrategy` class.
2. **Identical Logic**: `LegacyTargetStrategy` will perfectly encapsulate the existing `TargetFactory` `if/else` logic, ensuring dataset hashes generated for `NIFTY50/v3.0` remain mathematically identical.
3. **Inference Passthrough**: For legacy targets, `post_process_inference()` will simply return `raw_prediction`, ensuring no disruption to existing inference logic.

==========================================================
## SECTION 4 — IMPACT ANALYSIS
==========================================================
| Module | Impact Level | Justification |
|--------|--------------|---------------|
| **Feature Engineering** | NO CHANGE | Feature extraction operates independently and strictly prior to target generation. |
| **TargetFactory** | MAJOR CHANGE | Will be refactored into a `TargetStrategyRegistry` and multiple Strategy classes. |
| **TensorBuilder** | MINOR CHANGE | Updated to instantiate the strategy via the Registry instead of calling static Factory methods. |
| **Metadata** | MINOR CHANGE | Must record `strategy_name` into `metadata.json` so Inference knows how to post-process. |
| **Training Pipeline** | NO CHANGE | `TimeSeriesDataset` and PyTorch model logic remain completely target-agnostic. |
| **Inference Engine** | MINOR CHANGE | Must dynamically load the corresponding Strategy based on `metadata.json` and execute `post_process_inference()`. |
| **Registry** | NO CHANGE | The artifact bundle schema remains unchanged. |
| **Model Loader** | NO CHANGE | Standard PyTorch weight hydration remains identical. |

## SECTION 5 — IMPLEMENTATION PLAN
==========================================================
The implementation must occur in independently testable phases:

1. **Phase 1: Core Framework (No Pipeline Changes)**
   - Create `ml_engine/data/tensors/targets/base.py` (Base interface).
   - Create `ml_engine/data/tensors/targets/registry.py`.
   - Implement `LegacyTargetStrategy` encapsulating current logic.
   - *Test*: Unit test `LegacyTargetStrategy` output against raw dataframe operations.

2. **Phase 2: TensorBuilder Integration**
   - Update `TargetFactory` to act as a wrapper for `TargetStrategyRegistry` (for backward compatibility) or update `TensorBuilder` to use the registry directly.
   - Update `MetadataBuilder` to export `target_strategy` to `metadata.json`.
   - *Test*: Generate a tiny sample dataset and assert `y` arrays are identical to `main` branch.

3. **Phase 3: Inference Integration**
   - Update `InferenceEngine` to read `target_strategy` from `metadata.json`, load the strategy from the registry, and apply `post_process_inference()`.
   - *Test*: Mock an inference run and verify scalars pass through unmodified.

4. **Phase 4: New Target Expansion (Future)**
   - Create `VolatilityAdjustedStrategy`.
   - *Test*: Unit test volatility scaling and post-process un-scaling.

## SECTION 7 — RISK ANALYSIS
==========================================================
1. **Risk: Tensor Array Drift**
   - *Description*: Refactoring the mathematical implementation into a class introduces a bug that changes target array outputs by a few decimal places, silently corrupting training.
   - *Mitigation*: The `LegacyTargetStrategy` will literally copy-paste the exact Pandas vectorization logic currently in `TargetFactory`. A hash comparison test on `y_train.npy` will be mandatory.
   - *Rollback*: Revert `TensorBuilder` to import the original `TargetFactory`.

2. **Risk: Inference Metadata Missing**
   - *Description*: Models currently in production do not have a `target_strategy` key in their `metadata.json`.
   - *Mitigation*: `InferenceEngine` will use `dict.get("target_strategy", "legacy")` to safely default to the legacy passthrough strategy for all existing production models.
   - *Rollback*: Inference logic fails gracefully by returning the raw model scalar.
