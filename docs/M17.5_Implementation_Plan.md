# M17.5 Implementation Plan

==========================================================
## SECTION 1 — EXECUTIVE SUMMARY
==========================================================

**Confirmed Root Cause**: 
The `TimeSeriesDataset` class within `ml_engine/training/utils.py` unconditionally casts all target arrays (`y`) to `torch.long` (int64). 

**Why it affects regression**: 
Regression targets represent continuous fractional financial returns (e.g., `-0.015`, `0.026`). The `torch.long` cast forcibly truncates these small floating-point values down to integers, destroying the data by permanently converting all labels into exactly `0`. Consequently, the model collapses by correctly learning to output `0.0`.

**Why classification currently works**: 
Classification models (Binary and Multiclass) use integer class indices (e.g. `0`, `1`, `2`) as targets. Thus, truncating or casting them to integers (`torch.long`) is both correct and strictly required by classification loss functions such as `CrossEntropyLoss`.

**Why the fix is required**: 
Without resolving this hardcoded truncation, no regression model can be trained correctly on the production platform. Fixing this bug is a mandatory prerequisite for successful regression training.

==========================================================
## SECTION 2 — CURRENT ARCHITECTURE
==========================================================

The current runtime path executes as follows:

1. `train.pt` (Tensors loaded from disk)
   ↓
2. `TrainingOrchestrator._load_tensors()` (Converts true targets to NumPy `float64` array)
   ↓
3. `TimeSeriesDataset` (Constructs dataset and **truncates targets**: `self.y = torch.tensor(y, dtype=torch.long)`)  <-- **DESTRUCTIVE CAST HAPPENS HERE**
   ↓
4. `DataLoader` (Returns batches of `torch.int64` containing all zeros)
   ↓
5. `_train_epoch()` (Casts zeroed integers back to float: `y_batch = y_batch.float().to(device)`)
   ↓
6. `Loss` (`MSELoss` optimizes predictions against `0.0`)

## SECTION 3 — DESIGN GOALS
==========================================================

1. **Preserve Binary Classification behaviour**: `TimeSeriesDataset` will continue yielding `torch.long` for binary tasks.
2. **Preserve Multiclass Classification behaviour**: `TimeSeriesDataset` will continue yielding `torch.long` for multiclass tasks.
3. **Support Regression correctly**: `TimeSeriesDataset` will be updated to conditionally preserve `torch.float32` values.
4. **Support Multi-Output Regression correctly**: Will similarly preserve `torch.float32` target arrays.
5. **Require no runtime hacks**: We will extend the `TimeSeriesDataset` signature cleanly to accept an explicit `y_dtype` parameter.
6. **Keep TensorBuilder unchanged**: No data generation changes are necessary; existing arrays are already correctly formatted `float64` vectors.
7. **Keep metadata contract unchanged**: Metadata accurately reflects `return_5d` continuous values.
8. **Keep inference unchanged**: Inference does not use `TimeSeriesDataset` for labels.
9. **Maintain backward compatibility**: The new parameter on `TimeSeriesDataset` will default to `torch.long`, ensuring legacy scripts calling it without a dtype argument retain identical behavior.

## SECTION 6 — TASKTYPE HANDLING
==========================================================

- **BINARY_CLASSIFICATION**
  - **Required tensor dtype**: `torch.long`
  - **Reason**: Binary cross-entropy and general classification loss inputs demand integer indices representing positive/negative classes.
  - **Loss compatibility**: Compatible natively with PyTorch classification loss signatures.

- **MULTICLASS_CLASSIFICATION**
  - **Required tensor dtype**: `torch.long`
  - **Reason**: `CrossEntropyLoss` computes log-softmax internally and explicitly rejects fractional labels, requiring an integer class tensor.
  - **Loss compatibility**: Natively compatible.

- **REGRESSION**
  - **Required tensor dtype**: `torch.float32`
  - **Reason**: Fractional values must be preserved to compute distance metrics against raw float responses. 
  - **Loss compatibility**: PyTorch `MSELoss`, `HuberLoss`, and `L1Loss` mathematically require floating-point operands.

- **MULTI_OUTPUT_REGRESSION**
  - **Required tensor dtype**: `torch.float32`
  - **Reason**: Same as single-output regression; multivariate distance computations require floating point matrices.
  - **Loss compatibility**: Natively compatible with `MSELoss`.

## SECTION 9 — RISK ANALYSIS
==========================================================

- **Risk**: Circular Import dependencies if `TaskType` is imported into low-level `utils.py`.
  - **Likelihood**: High (if implemented incorrectly).
  - **Impact**: High (ImportError crashing the pipeline).
  - **Mitigation**: Implemented via parameter injection. We avoid passing `TaskType` directly into `utils.py`. Instead, we pass `y_dtype=torch.float32` from the Orchestrator, which already possesses the required enum dependencies.

- **Risk**: GPU Device Mismatch.
  - **Likelihood**: Low.
  - **Impact**: High (Runtime crash during loss calculation).
  - **Mitigation**: `_train_epoch()` in `training_pipeline.py` already includes explicit `.to(device)` mapping on both `X_batch` and `y_batch`, shielding the dataset-level dtype shift.
