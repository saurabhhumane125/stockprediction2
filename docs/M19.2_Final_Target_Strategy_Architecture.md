# M19.2 Final Target Strategy Architecture

==========================================================
## OBJECTIVE
==========================================================
Finalize the production architecture for the Target Strategy Framework. This design eliminates hardcoded target logic, enforces the Open/Closed Principle, separates target generation from prediction decoding, and guarantees perfect backward compatibility for existing production models.

==========================================================
## 1. TARGET MANAGER
==========================================================
The `TargetManager` acts as the single authoritative dependency for resolving target strategies. 

**Responsibilities**:
- **Loading**: Scans the strategy registry (or directory) and initializes registered strategy classes.
- **Validating**: Ensures every loaded strategy complies with the `BaseTargetStrategy` interface and specifies a valid version.
- **Exposing**: Provides a resolved strategy instance to consumers (like `TensorBuilder`) based on the requested `TargetConfig`.

`TensorBuilder` will **not** instantiate or discover strategies. It will be injected with (or request) a resolved strategy instance from the `TargetManager`.

==========================================================
## 2. RESPONSIBILITY SEPARATION
==========================================================
The architecture rigidly enforces the separation of **Target Generation** (data preparation) and **Inference Decoding** (prediction translation).

- **TargetStrategy**: Owns **generation only**. It calculates mathematical targets from historical price data (e.g., `(close_{t+5} - close_t) / close_t`) and defines what column names it outputs. It does **not** touch predictions.
- **InferenceDecoder**: Owns **decoding only**. At inference time, the raw tensor output of the PyTorch model is passed to an `InferenceDecoder` (resolved via `TargetManager` or a dedicated `DecoderRegistry`), which translates the raw scalar back into human-readable business logic using metadata and current features.

==========================================================
## 3. STRATEGY VERSIONING
==========================================================
Every strategy class strictly defines a semantic version. If a mathematical formula for a target changes in the future, a new version is created (e.g., `1.1`), while the old version remains immutable in the codebase.

```python
class BaseTargetStrategy(ABC):
    @property
    @abstractmethod
    def strategy_name(self) -> str: pass

    @property
    @abstractmethod
    def strategy_version(self) -> str: pass
```
*Example*: `strategy_name = "raw_return"`, `strategy_version = "1.0"`.

==========================================================
## 4. METADATA CONTRACT
==========================================================
To ensure the Inference Engine can flawlessly decode predictions from models trained months ago, the `metadata.json` must capture the complete target specification immutable at training time.

**Schema Additions**:
- `strategy_name` (str): Used by the `TargetManager` during inference to route to the correct `InferenceDecoder`.
- `strategy_version` (str): Ensures the exact mathematical decoding logic is applied, avoiding backward-breaking updates.
- `task_type` (str): "REGRESSION", "BINARY_CLASSIFICATION" etc. Tells the core inference engine the shape and loss context of the model output.
- `target_type` (str): Secondary classification (e.g. "CLASS", "LOG_RETURN") for legacy dataset routing.
- `primary_horizon` (int): The main lookahead window (e.g., 5). Required to label the prediction correctly in the UI/Response.
- `forecast_horizons` (List[int]): Required for multi-output regression to know which index maps to which horizon.
- `target_columns` (List[str]): The exact column names exported to the tensor (e.g., `["return_5d"]`). Verifies tensor dimensions.
- `output_schema` (Dict): Defines the expected semantic range (e.g., `{"min": -1.0, "max": 1.0, "dtype": "float32"}`) for outlier detection at inference time.

==========================================================
## 5. CONFIGURATION CONTRACT
==========================================================
Switching target strategies is executed strictly via `config/training_config.py`.

```python
class TargetConfig(BaseModel):
    strategy_name: str = "legacy"  # "raw_return", "vol_adjusted", etc.
    strategy_version: str = "1.0"
    task_type: TaskType = TaskType.REGRESSION
    # ... legacy fields ...
```
No source code edits will be required to switch between mathematically distinct target generation pipelines.

==========================================================
## 6. PIPELINE OWNERSHIP
==========================================================
- **TargetManager**: Owns discovery, validation, and instantiation of strategies and decoders.
- **TargetStrategy**: Owns the mathematical formula that converts a DataFrame of features into a target column `y`.
- **TensorBuilder**: Owns disk I/O. It consumes a `TargetStrategy`, applies it to the DataFrame, extracts the numpy arrays, and saves `.npz` files.
- **MetadataBuilder**: Owns serializing the resolved `TargetStrategy`'s properties into `metadata.json`.
- **TrainingPipeline**: Owns the PyTorch training loop. It blindly optimizes for the `y` array provided, completely unaware of the target strategy.
- **InferenceDecoder**: Owns the mathematical formula that converts a raw model scalar into a business prediction (e.g., `raw_pred * current_volatility`).
- **Registry**: Owns the immutable storage of the model state and `metadata.json`.
- **ModelLoader**: Owns loading PyTorch weights into memory.

==========================================================
## 7. DEPENDENCY GRAPH
==========================================================
```text
Config -> TargetManager

DatasetGenerator -> TensorBuilder
TensorBuilder -> TargetManager (Requests Strategy)
TargetManager -> TargetStrategy (Returns Instance)
TensorBuilder -> TargetStrategy (Executes logic)

TensorBuilder -> MetadataBuilder (Passes strategy properties)

TrainingPipeline -> (Reads .npz files from disk)
TrainingPipeline -> Registry (Uploads model.pt and metadata.json)

InferenceEngine -> ModelLoader (Loads model.pt)
InferenceEngine -> TargetManager (Requests InferenceDecoder via metadata.json)
TargetManager -> InferenceDecoder (Returns Instance)
InferenceEngine -> InferenceDecoder (Passes scalar for translation)
```
*No circular dependencies. Information flows strictly forward.*

==========================================================
## 8. EXTENSION MODEL
==========================================================
To add `VolatilityAdjustedReturn` (version `1.0`):

1. **Create Target Strategy**: `ml_engine/data/tensors/targets/strategies/vol_adjusted.py`.
   - Implements `BaseTargetStrategy`.
   - Defines `strategy_name = "vol_adjusted"`, `strategy_version = "1.0"`.
   - Implements `generate()` to divide returns by volatility.
2. **Create Inference Decoder**: `ml_engine/inference/decoders/vol_adjusted_decoder.py`.
   - Implements `BaseInferenceDecoder`.
   - Maps to `strategy_name = "vol_adjusted"`.
   - Implements `decode()` to multiply the raw scalar by current volatility.
3. **Register**: Register both in `TargetManager`.
4. **Configure**: Update `TrainingConfig.target.strategy_name = "vol_adjusted"`.

*Zero existing files are modified to support this new mathematical logic.*

==========================================================
## 9. MIGRATION STRATEGY
==========================================================
**Guaranteeing Existing Models Work**:
- **Existing Metadata**: Production models in the registry lack the `strategy_name` key in their `metadata.json`.
- **Inference Migration**: `TargetManager.get_decoder(metadata)` will implement a fallback:
  ```python
  strategy = metadata.get("strategy_name", "legacy")
  ```
- **Legacy Strategy**: A `LegacyTargetStrategy` and `LegacyInferenceDecoder` will perfectly replicate the existing hardcoded logic.
- **Existing Datasets/Tensors**: Remain completely valid. The legacy target logic produced them, and the legacy decoder translates them. No regeneration required.
- **Existing Registry**: No artifacts need modification.

## 12. RISK REVIEW
==========================================================
**Risk 1: Metadata Missing on Old Models (Inference Failure)**
- *Review*: If the inference engine hard-requires `strategy_name`, old models will crash.
- *Resolution*: The architecture explicitly dictates a fallback to `"legacy"` if `strategy_name` is missing. This is locked into the design.

**Risk 2: Dataset Generation Hash Mismatch**
- *Review*: Moving logic into `LegacyTargetStrategy` might unintentionally alter float precision or NaN dropping logic, causing dataset drift.
- *Resolution*: The implementation of `LegacyTargetStrategy.generate()` must be a literal copy-paste of the legacy code. An automated regression test will hash the output tensor to guarantee mathematical equivalence.

**Risk 3: Model Target Leakage**
- *Review*: If a strategy requires current volatility to decode, does the inference payload contain current volatility?
- *Resolution*: Yes, `BaseInferenceDecoder.decode(raw_prediction, current_features)` explicitly takes `current_features` (a row from the inference DataFrame) ensuring the decoder has access to all latest feature vectors (including volatility) without re-calculating them.

**Conclusion**: The architecture satisfies all SOLID principles, eliminates hardcoded logic, completely separates generation from inference, and guarantees backward compatibility for all existing ML assets. It is approved for implementation.
