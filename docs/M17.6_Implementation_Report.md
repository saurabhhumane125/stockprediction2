## 3. Why each change was required
==========================================================
- The change in `utils.py` was strictly necessary to unblock the dataset from destructively casting unscaled float responses (financial returns) to 0. 
- The change in `training_pipeline.py` ensures the orchestrator actively takes responsibility for dtype injection based on `TaskType`, avoiding assumptions. 
- The explicit comment addresses the design refinement request, highlighting that although `utils.py` contains a fallback for older scripts, the pipeline itself is fully explicit.
- The unit tests provide safety nets preventing future regressions to this core data structure behavior.

==========================================================
## 4. How backward compatibility was preserved
==========================================================
We preserved backward compatibility through two deliberate mechanisms:
1. **At the Dataset level**: `TimeSeriesDataset.__init__` handles omitted arguments by defaulting to `torch.long`. Thus, any legacy scripts that construct it identically to before will experience no change.
2. **At the Pipeline level**: `_build_loaders()` manually applies the same fallback (`torch.long`) when the orchestrated `TaskType` belongs to classification (binary or multiclass), guaranteeing that their evaluation matrices and `CrossEntropyLoss` operands remain identical to historical runs.
