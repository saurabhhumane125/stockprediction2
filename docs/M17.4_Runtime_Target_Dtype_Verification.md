## SECTION 5 — SOURCE CODE TRACE
==========================================================

**Occurrence 1: `ml_engine/training/utils.py`**
- **Type**: `torch.long`
- **Purpose**: Instantiates the `y` tensor inside `TimeSeriesDataset` (Line 77). 
- **Task Type**: Universal (Agnostic to TaskType, applied blindly).
- **Affects regression?**: Yes. It destroys all fractional regression targets by integer truncation.

**Occurrence 2: `ml_engine/training/training_pipeline.py`**
- **Type**: `torch.float`
- **Purpose**: Casts `y_batch` to a float immediately before passing it to `criterion` during `_train_epoch()` (Line 437).
- **Task Type**: `TaskType.REGRESSION` and `TaskType.MULTI_OUTPUT_REGRESSION`.
- **Affects regression?**: Yes. It converts the destroyed integer targets (`0`) into floating zeroes (`0.0`) so the MSE loss function doesn't crash on integer types.

==========================================================
## SECTION 6 — CLASSIFICATION COMPATIBILITY
==========================================================

**Why does TimeSeriesDataset currently use `torch.long`?**
- **Binary Classification**: Required. PyTorch `CrossEntropyLoss` and standard classification criteria expect integer class indices (0 or 1) of type `torch.long`.
- **Multiclass Classification**: Required. Multiclass CrossEntropy expects integer class indices (0, 1, 2) of type `torch.long`.
- **Regression**: **Incompatible**. Regression requires continuous floats (e.g. `torch.float32`). The blind application of `torch.long` across all TaskTypes causes the truncation bug.

==========================================================
## SECTION 7 — ROOT CAUSE VERDICT
==========================================================

**CONFIRMED**

Regression targets are destroyed by integer casting.

**Support**: Runtime execution tracing proves that original `float64` labels holding the fractional values are ingested by `TimeSeriesDataset` and cast indiscriminately to `torch.long` (int64). This truncates values like `0.0262` directly to `0`. By the time `_train_epoch()` executes, the loss function attempts to minimize distance to a constant array of exactly `0.0`.

==========================================================
## SECTION 8 — IMPLEMENTATION IMPACT
==========================================================

- **Files affected**: `ml_engine/training/utils.py`
- **Classes affected**: `TimeSeriesDataset`
- **Functions affected**: `__init__()`
- **Implementation Scope**: `TimeSeriesDataset` must become `TaskType` aware (or at least dtype-aware) so it dynamically casts `y` to `torch.float32` for continuous tasks, and `torch.long` for classification tasks, preserving the fractional targets.
- **Risk assessment**: Low risk if gated by task type. 
- **Regression impact**: Critical fix. Resolves model collapse, enabling regression training to function.
- **Classification impact**: None, provided the legacy behavior (`torch.long`) is preserved for `BINARY_CLASSIFICATION` and `MULTICLASS_CLASSIFICATION`.
